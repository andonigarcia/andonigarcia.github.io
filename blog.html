Responsive Design

3 October 2014

Last time I ended my blog post talking about how frustrating it is to design a website and then have the layout be completely messed up everytime you shorten the screen width. The solution to this is a practice called Responsive Design.

I guess I didn't do my homework well enough, because I found out about responsive design only after taking to Twitter Bootstrap, a very elegant and widely used framework and templating for Web Design. I know I was supposed to learn Javascript after CSS, but this bugged me so much, that I decided it needed to come first. I had been hearing about Bootstrap for a long time, but wanted to learn the basics first and then build upwards to frameworks (such a UChicago mindset).

After reading through the Bootstrap documentation and information, I finally realized what responsive mobile-first design is. The main idea is to design your website to be ideal on the smallest screen first. So mobile-device screen widths and smaller have a pleasant UX. Only after that, do you start to think about and design for larger screens. You use media queries such as @media (min-width: 720px) to specify how your CSS layout should appear for larger screens. But it should first and foremost be optimized for small screens. If you do want to display additional or different items on larger screens that don't appear on smaller screens, use display: none appropriately on those elements' CSS. Finally, make sure that your width's and lengths are all in relative terms. In stead of specifying a div's width to 250px, make it 15%. This will allow it to appropriately scale as screen widths are adjusted! Height isn't as big of a deal, but width is essential.

Luckily, Bootstrap has responsive design embedded and intertwined into its language. A navbar automatically condenses and dropdowns respond accordingly. It makes your life so much easier. Bootsrap 3 was finally made mobile-first, so all of its classes and components that come with the library are enhanced for a nice responsive design. I'd ultimately like to next design a website that is responsive all on my own, but I'll need Javascript for the dropdowns and other things. Practically the only components of my site that are Bootstrap are the Navbar and Footer. But Bootstrap was a very very nice deviation from my learning path, because it finally allowed me to be extremely happy with how my website turned out and allowed me to have a nice and responsive website with little effort.

I strongly encourage you all to go through and read the Bootstrap documentation thoroughly before using it. It will take one afternoon and after you will have an excellent grasp on the ease of using Bootstrap to dramatically enhance your website. It's a great learning tool to help fine-tune your CSS skills and you will really appreciate it. Just download the files and you'll be all set! No installation or other crazy downloads necessary. Literally just grab their CSS and JS files. Cheers. Next time, I'll be coming to you with some Javascript knowledge hopefully!

-Andoni M. Garcia

Next Came CSS

2 October 2014

HTML is only so helpful. Sure you can "write" a webpage, but you definitely cannot make it look like a webpage at all. Lucky for me, Codecademy bundled the two together. As such, while learning HTML5 I also got a nice little intro into CSS3. While it wasn't at all a comprehensive intro, it was very nice to understand how CSS interacted with my HTML. Also props to Codecademy for putting a large emphasis on <div> tags. These have BY FAR been the most helpful and most crucial things to learn for designing a nice UX. Everything from a navigation bar and a footer (at least most of the time) to where you display your content relies on div tags.

Of course this intro couldn't have hit on everything. But did it need to? Unlike the various key HTML elements (like <nav> <iframe> or <video> tags) that Codecademy skipped, they didn't really skip any essential CSS properties. I have yet (but eventually plan to) read the Mozilla Developer Network documentation for the CSS3 properties. MDN does a great job at making comprehensive documentation for HTML5/CSS3, but there are just too many CSS properties to reasonable read through all of them before getting my feet wet. Thus all the CSS code I used to program my first two websites, I entirely learned from Codecademy.

This CSS knowledge was adequate. My first website I had a little navbar at the top, nothing too fancy or special. The dropdown was kinda clunky and not the best UX design, but it worked. I also had a sidebar that, while barren, nicely closed the page. For my second site, I had a slightly nicer navbar (arguably), incorporated text overlay on images, and the biggest achievement: Multiple Pages. The one thing that Codecamey failed to teach me was how to paginate my website.

I casually breezed through the videos by the same guy that I referenced for learning a solid path to becoming a Web Developer. These videos didn't really teach me anything after having gone through the Codecademy tutorial and tinkering around with my own website. However, hidden in one of the videos, I found out something that had always escaped me: How to make multiple pages for your website.

If you are like me and just starting off, here's the tip: Make another HTML file, say page2.html, and code it as if you were making a new webpage entirely. Add the CSS for this new HTML file in your typical main.css (unless you want to link a bunch of CSS files, which may be helpful depending on the type of website you are building. If the style changes depending on the part of the site, you may want to separate the CSS files, but for me, I keep everything in main.css for now). Then in your index.html, wherever you want to reference the other page, simply type <a href="page2.html">. It really is that easy. The browser will first look in your current working directory for that file, page2.html, so you don't have to specify a path, an http, or anything. Just the file name. Much like linking images.

This was super helpful and allowed me to finally design a nice and impressive Website rather than just a simple Webpage. However, I still wasn't satisfied. I often work split screen, with my browser condensed to the left and my editor to the right, each taking up 50% of the screen. Doing this though fucked up my layout. Everything got shifted and messed up and didn't look nice. I had a website, but it only delivered the proper UX if seen on fullscreen. And it definitely would not work on a phone. This bothered me tremendously. As such, I next looked into it, and found there is a whole subset of Web Development called responsive design. This then became the highest precedence on my path to becoming a Web Developer.

-Andoni M. Garcia

Starting With HTML

1 October 2014

Clearly the first thing you need to learn to be a Web Developer is how to write HTML. I started this journey by toying around with Codecademy. This ended up being a really helpful start to my learning. Not because it taught me everything I needed to know. Not even close. But because it offers an interactive and easy way to clearly see what you're doing and get a good grasp on how to use HTML initially. It taught me the basics and got me familiar with using headers, paragraphs, divs, etc. This was a good start and allowed me to toy around with my first website (kinda. I had to also take the CSS course, but more on that on the next blog entry).

After toying around with Codecademy and getting a good grasp of designing a website and the programming behind it, I wanted to learn the actual language of HTML itself. Clearly there were elements and attributes that Codecademy breezed over or skipped in its quick (one afternoon of learning) intro. After looking more into it, I found the Mozilla Developer Network to have really good documentation of HTML5.

As such, I spent the next couple of days going through each element's page, reading its attributes and understanding it's functionality. This definitely prepared me and made me confident in my physical HTML skills. Next was to learn CSS and build my first website!

-Andoni M. Garcia

My Path To Learning Web Development

1 October 2014

Near the end of this summer, I finally decided to learn how to program the Web once and for all. After taking a few Codecademy courses on HTML, CSS and Javascript/JQuery, I wanted to know the full path of the endeavor I was embarking on. The more I read about Web Development the more confused I got.

What is a framework? What is back-end programming? How does Python/C++/languages I've learned through school apply to web programming? What is bootstrap and how do I use it? SASS? LESS? Are all of these things necessary? It seemed like there were a ton of different languages and libraries I had to learn just to code a website and make it do something. I knew there had to be a good explanation of all these things and how they fit together. But it was so hard to find something that easily explained Web Development.

Then, I stumbled on this video, which now shapes and directs how I learn and approach Web Development:

How to Become a Web Developer

Afterwards, I developed a schedule of how I would approach learning Web Development. I began with the basics, added in some complexities, and started with Front-End progressing to Back-End. I find it critical to learn the absolute basics first and foremost before learning templates and tools. In that regard, I fiddled with websites sin any frameworks before coming to this Bootstrap site. In a later blogpost I will explain my previous endeavors. For now, this is my schedule:

  • Basic Front End
    • HTML5
    • CSS3
    • Basic Javascript
  • Javascript Frameworks & Libraries
    • JQuery & Ajax
    • Angular.js OR Backbone.js
  • CSS Frameworks & Libraries
    • SASS & LESS
    • Bootstrap
  • Basic Back End
    • Ruby & Ruby on Rails
    • Python & Django
    • Node.js & Express.js
    • PHP
  • Databases
    • MySQL
    • MongoDB
  • Caching
    • Memcached OR Nginx
  • Advanced Back End
    • Security/OAUTH2
    • Authorization/Authentication
    • API's

I might do some of this out of order (like learning Bootstrap before SASS/LESS or JS) and other such things, but mostly I want to stick to this general outline of learning. If you are in my boat and learning to WebDev and have questions or are an advanced and experienced developer and want to give advice or suggestions, feel free to reach out and contact me! I'm a very chill dude and would love to bestow my knowledge on someone else or receive helpful tips, pointers, and suggestions!

-Andoni M. Garcia
